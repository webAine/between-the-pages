## 2026-05-17 — Базовая сцена и ресурсы

### main.tscn + main.gd
Создана главная сцена. Структура:
```
Main (Node2D) ← main.gd
├── Background (ColorRect) — тёмно-коричневый фон
├── ShopRoom (Node2D) — контейнер комнаты
│   ├── ShelvesArea → ShelvesContainer (HBoxContainer) — горизонтальные шкафы
│   ├── Counter → CatalogSpot — прилавок и место каталога
│   └── CustomerArea — зона покупателя
└── UI (CanvasLayer)
    └── HUD → MoneyLabel, ReputationLabel, OrderLabel
```

`main.gd` хранит три ресурса: `money`, `reputation`, `shelf_order`.
Методы: `change_money()`, `change_reputation()`, `change_order()` — каждый обновляет HUD через `update_hud()`.
`clampi()` не даёт репутации и порядку выйти за 0–100.

### Book (scripts/book.gd)
Resource-класс для одной книги.
Поля: название, автор, жанр, описание, цвет обложки, толщина, расположение на полке, наличие.
Enum-ы: `Genre`, `Thickness`, `ShelfRow`, `ShelfSpot`.
Статические методы `genre_to_string()`, `thickness_to_string()` и т.д. — для отображения в UI.
`_parse_genre()` — парсит строку из JSON в enum (используется и Catalog, и Customer).

### Catalog (scripts/catalog.gd)
Resource-класс: массив `Array[Book]`.
Методы: `find_by_title()`, `find_by_genre()`.
`load_from_file(path)` — читает `data/books.json` и возвращает заполненный Catalog.

## 2026-05-18 — Покупатели

### Customer (scripts/customer.gd)
Resource-класс для покупателя.
Три уровня (`tier`): `STORY`, `REGULAR`, `RANDOM`.
Три типа запроса (`request_type`): `KNOWS_TITLE`, `KNOWS_GENRE`, `DOESNT_KNOW`.
Поля диалогов: `dialogue_enter`, `dialogue_wait`, `dialogue_correct`, `dialogue_wrong`.

`KNOWS_TITLE` — знает точное название книги.
`KNOWS_GENRE` — называет жанр, игрок выбирает из нескольких.
`DOESNT_KNOW` — только подсказки внешности/поведения (`visual_hints`), целевой жанр скрыт.

`load_all_from_file(path)` — читает `data/customers.json`, возвращает `Array[Customer]`.

### DayManager (scripts/day_manager.gd)
Node-класс (не Resource — это живая логика, не данные).
Управляет ходом одного игрового дня.

**Сигналы** (способ сообщать main.gd о событиях без прямой связи):
- `customer_arrived(customer)` — пришёл новый покупатель
- `customer_left(customer, satisfied, money_delta)` — покупатель ушёл
- `day_ended(day)` — все покупатели дня обслужены

**Как собирается очередь дня (`_build_queue`):**
1. Проверяем каждого REGULAR — пора ли ему по расписанию
2. Добираем RANDOM-покупателей из перемешанного пула до MAX (5 в день)
3. Перемешиваем итоговую очередь

**Цены книг:**
- Тонкая: 8
- Средняя: 12
- Толстая: 18

**Проверка правильности книги (`_check_book`):**
- `KNOWS_TITLE` → точное совпадение названия
- `KNOWS_GENRE` → совпадение жанра
- `DOESNT_KNOW` → совпадение с `target_genre` покупателя

**Репутация:**
- Правильная книга: +5
- Неправильная: -3

**Связь с main.gd:**
`DayManager` создаётся в `_ready()`, `main.gd` подписывается на его сигналы.
Сейчас реакция на события — только `print()` в консоль (визуал ещё не сделан).

## 2026-05-19 — UI и визуальные полки

### CustomerDialogue → main.gd
CustomerDialogue подключён к игровому циклу.

**Поток:**
1. `day_manager.customer_arrived` → `_dialogue.show_customer(customer)` — показывает панель с именем, репликой, запросом
2. Игрок нажимает "Принести книгу" → открывается каталог
3. Игрок нажимает "Принести не ту" → `_find_wrong_book()` → `resolve_visit()` (отладочная кнопка, потом уберём)
4. `day_manager.customer_left` → `_dialogue.show_result(customer, satisfied)` — показывает реакцию и кнопку "Следующий →"
5. "Следующий →" → `day_manager.next_customer()`

### Background — исправление
ColorRect Background перемещён из Main (Node2D) в UI (CanvasLayer) и поставлен **первым** дочерним нодом (перед HUD).
Причина: Control нод внутри Node2D не имеет правильного родительского прямоугольника для якорей. Внутри CanvasLayer — работает корректно.
Layout → Full Rect растягивает Background на весь экран автоматически.

Разрешение проекта: **1280×720** (стандартное HD).

### CatalogUI (scripts/catalog_ui.gd)
CanvasLayer (layer=3) — рендерится поверх всего.

**Структура панели (900×520, по центру экрана):**
- Левая колонка (170px) — кнопки жанров (7 штук)
- Центральная колонка (340px) — ScrollContainer со списком книг выбранного жанра
- Правая колонка (330px) — карточка книги (скрыта пока не выбрана)

**Карточка книги содержит:**
- Цветной прямоугольник обложки
- Название, автор
- Жанр + толщина
- Описание сюжета
- Физическое расположение: "Шкаф N, средний ряд, начало"

**Важное решение:** кнопка "Взять эту книгу" убрана. Каталог — справочник, не инструмент выбора. Книгу берут только с полки.

Открывается по нажатию "Принести книгу" в диалоге покупателя. Закрывается кнопкой ✕.

### ShelfDisplay (scripts/shelf_display.gd)
CanvasLayer (layer=1) — рендерится под диалогом и каталогом.

**Структура:**
- Тёмный фон (520px высота, оставляет место для диалоговой панели внизу)
- ScrollContainer с горизонтальной прокруткой
- 9 шкафов в HBoxContainer, каждый 200px шириной
- В каждом шкафу 3 ряда (TOP/MIDDLE/BOTTOM) с деревянной полкой
- Книги — цветные прямоугольники по цвету обложки, ширина зависит от толщины (16/26/40px)
- Клик на книгу → сигнал `book_clicked(book)` → `day_manager.resolve_visit(book)`

**Геймплейный поток:**
1. Покупатель называет запрос
2. Игрок открывает каталог → находит книгу → читает "Шкаф 3, средний ряд, начало"
3. Закрывает каталог → находит шкаф 3 → средний ряд → кликает на книгу нужного цвета и толщины
4. DayManager проверяет правильность → покупатель реагирует

## 2026-05-19 — Полировка геймплея и рефакторинг UI

### Убрана кнопка "Принести не ту"
Отладочная кнопка удалена из CustomerDialogue и из main.gd (`_on_bring_wrong_book`, `_find_wrong_book`).
Ошибка теперь должна быть результатом реального неправильного выбора на полке, а не кнопки.

### Исправлен клиппинг шкафов
Добавлено `clip_contents = true` на каждый cabinet ColorRect.
Причина: без этого текст и книги из одного шкафа визуально "вытекали" в соседний при прокрутке.

### Слоёвая система — итоговое решение
Порядок слоёв:
- `ShelfDisplay` (CanvasLayer, layer=1) — полки
- `CustomerDialogue` (CanvasLayer, layer=2) — панель покупателя
- `CatalogUI` (CanvasLayer, layer=3) — каталог
- `DayResultsUI` (CanvasLayer, layer=4) — экран итогов
- `UI` (CanvasLayer в сцене, layer=2) — HUD (деньги/репутация/порядок)

HUD нод (`Control`, FULL_RECT) получил `mouse_filter = 2` (Ignore) чтобы не блокировать клики на полках.

### DayResultsUI (scripts/day_results_ui.gd)
Экран итогов дня. Показывается когда все покупатели дня обслужены.

**Содержит:**
- Номер дня
- Счёт довольных покупателей (N из M)
- Заработано монет за день
- Изменение репутации за день
- Кнопка "Следующий день →"

Сигнал `next_day_pressed` → main.gd сбрасывает дневные счётчики, запускает новый день.

**Счётчики в main.gd:** `_day_money_earned`, `_day_satisfied`, `_day_total`, `_day_reputation_delta` — накапливаются в `_on_customer_left`, сбрасываются в `_on_next_day`.

### Новый layout UI — правая панель
Переработан общий макет экрана:

```
┌──────────────────────────────┬────────────┐
│                              │  [Портрет] │
│      Полки (960px)           │  Имя       │
│      горизонтальный скролл   │  Диалог    │
│                              │  Запрос    │
│                              │            │
│                              │  [Кнопка]  │
│                              │   320px    │
└──────────────────────────────┴────────────┘
```

**CustomerDialogue** переделан из нижней полосы в правую панель (960, 0) размером 320×720:
- Всегда виден как структурный элемент
- Содержимое скрыто когда нет покупателя (`_set_content_visible(false)`)
- Портрет-заглушка (ColorRect) сверху — под будущий арт
- "Открыть каталог" заменило "Принести книгу"

**ShelfDisplay** сужен с 1280px до 960px, высота увеличена до 720px (весь экран).

## 2026-05-19 — Механика поиска: инспекция, смещение книг, состояние

### BookInspectionUI (scripts/book_inspection_ui.gd)
Новый файл. CanvasLayer layer=1.

Клик на книгу больше не отдаёт её сразу. Открывается панель (400×300, по центру полок) с:
- Цветной прямоугольник обложки
- Название и автор (или "название неразборчиво" если книга потёртая)
- Жанр · толщина
- Состояние (новая / читаная / потёртая)
- Декор (золотой орнамент / тиснение — если есть)
- Кнопка "Отдать покупателю" (недоступна без активного покупателя)
- Кнопка "Положить обратно"

Сигналы: `book_confirmed(book)` → main.gd вызывает `resolve_visit`, `book_returned` → просто закрывает.

### ShelfDisplay — смещение книг (scripts/shelf_display.gd)
Добавлена логика случайного смещения книг. При вызове `setup()` и `refresh()` генерируется словарь `_actual_positions` — реальные позиции книг на полке, которые могут отличаться от данных каталога.

Процент смещения передаётся из main.gd через `_displacement_percent()`:
- shelf_order 0–30 → 40% книг не на месте
- shelf_order 31–60 → 20%
- shelf_order 61–100 → 5%

При переходе к новому дню вызывается `refresh()` — позиции пересчитываются.
Каталог всегда показывает официальное место из данных. Расхождение — это и есть геймплей.

### ShelfDisplay — переименование шкафов (scripts/shelf_display.gd)
Заголовки шкафов заменены с Label на LineEdit (editable=false). Клик → поле становится редактируемым. Enter или потеря фокуса → сохраняет в `_cabinet_names[i]`. Названия переживают смену дней (пересборка шкафов читает из того же массива).

### Book — поля состояния и декора (scripts/book.gd)

## Текущее состояние

Полный игровой цикл с расширенной механикой поиска:
- Покупатель приходит → правая панель показывает портрет, диалог, запрос (4 типа)
- Игрок открывает каталог → находит книгу → видит состояние, декор, расположение
- Игрок ищет книгу на полке (может быть не там где каталог говорит)
- Клик → панель инспекции: проверяет состояние, видит читаемо ли название
- "Отдать" или "Положить обратно" → покупатель реагирует
- День заканчивается → экран итогов → следующий день (полки пересчитываются)
